# EOE Architecture Executable (AdminG)

## Scope
This document consolidates the current system as an operational EOE without breaking existing modules.

## Current Active Base (kept as-is)
- Operational Units/Operations
- CRM
- Quick Settings
- POS (partial)
- AI assistant
- Notifications
- WhatsApp (partial)
- Multi-tenant, roles, and plans

## Non-ambiguous classification
Each component has 3 roles (module, business type, entry point), with one dominant role.

| Component | Module | Dominant Business Type | Entry Point |
|---|---|---|---|
| Publicidad | yes | agencia_marketing | captacion_usuarios |
| Cobros | yes | gestion_cobros_recaudo | pago_deuda |
| Consultorias | yes | servicios_profesionales | agenda_contacto |
| POS | yes | tienda_retail | caja_punto_fisico |
| Nomina | yes | gestion_laboral_rrhh | pago_gestion_empleados |
| Auditoria | yes | auditoria_control | revision_inspeccion |
| Gobernanza/Politica | yes | institucional_publico | proceso_institucional |
| Quick Settings | yes | n/a | interno |
| IA | yes | n/a | interfaz_inteligente |
| OmniRisk | latent | n/a | n/a |

## Runtime flow
Entry Point -> Business Type -> Modules -> Operational Unit -> Event -> Audit -> Payroll

## Governance (active)
Checks applied at execution time:
- permissions
- context
- limits by plan

## Traceability and audit (active)
Events and logs allow reconstruction of:
- operations
- charges
- payroll calculations
- decisions

## OmniRisk (latent, prepared)
No complex logic enabled yet. Prepared with:
- enriched event payloads (context, actor, impact)
- hooks and fixed risk taxonomy
- placeholder endpoint `/risk/evaluate`

## Executable endpoints added
- `POST /execute-operational-unit`
- `POST /financial/charge`
- `POST /payroll/calculate`
- `GET /audit/logs`
- `POST /risk/evaluate` (placeholder)
- `GET /eoe/classification`
- `GET /eoe/business/declarations`

## Practical integration
- POS/charges use payment records and can generate cash transactions when source module is POS.
- Payroll calculation uses real traces:
  - operational events executed by employee
  - sales/payments generated by employee
  - service-linked payments generated by employee
- Optional persistence of payroll result to `payroll_payments`.

## Security and tenancy
- Token user resolution with tenant scope resolution.
- Owner/scope enforcement on operations, charges, payroll, and audit logs.

## Notes
- This is an operational baseline, not a full OmniRisk implementation.
- Keeps complexity low and reuses existing modules to avoid duplicated logic.
